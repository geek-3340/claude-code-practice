## 1. サブエージェントとは

サブエージェントは、特定の専門タスクに特化した「助手」です。
メインの Claude Code（リードエージェント）が必要に応じて呼び出します。

### メインエージェントとの違い

| | メインエージェント | サブエージェント |
|---|---|---|
| コンテキスト | 全体の会話を保持 | 独立した 200K トークンのコンテキスト |
| 役割 | 全体の指揮・調整 | 特定の専門タスクを実行 |
| 起動方法 | `claude` で起動 | メインから自動で呼び出される or `/agents` |

### サブエージェントのメリット

**① コンテキストの分離**
メインの会話がサブエージェントのタスクで汚れない。
情報収集のためにファイルを大量に読んでも、メインのコンテキストは消費されません。

**② 専門性の担保**
「リサーチ専門」「テスト専門」「レビュー専門」のように、
それぞれに最適化された指示を事前に定義できます。

**③ 並列処理**
複数のサブエージェントを同時に走らせて、作業を並列化できます。

## 2. サブエージェントの定義方法

### 方法1：`/agents` コマンド

```
/agents
```

対話的にサブエージェントを作成できます。

### 方法2：ファイルを直接作成

```bash
# プロジェクト用（チームで共有）
mkdir -p .claude/agents
nano .claude/agents/researcher.md

# 個人用（全プロジェクトで使用）
mkdir -p ~/.claude/agents
nano ~/.claude/agents/reviewer.md
```

### サブエージェント定義ファイルの書き方

```markdown
---
description: "コードベースの調査・情報収集を行う専門エージェント"
allowed-tools:
  - read
  - grep
  - glob
  - web_search
---

# リサーチャー

あなたはコードベースの調査を専門とするエージェントです。

## 振る舞い
- 依頼されたトピックに関連するファイルを徹底的に探索する
- 発見した情報を構造化して報告する
- コードの変更は行わない（読み取り専用）

## 出力フォーマット
- 関連ファイルの一覧（パスと概要）
- 重要な関数・クラスの説明
- 依存関係の整理
```

**ポイント：** `allowed-tools` で使えるツールを制限できます。
リサーチ用なら読み取り系のみ、実装用なら書き込みも許可、という使い分けが可能です。

## 3. 実用的なサブエージェント例

### ① リサーチャー — 情報収集専門

```markdown
---
description: "コードベースの調査と情報整理"
allowed-tools:
  - read
  - grep
  - glob
  - web_search
---

# リサーチャー
依頼されたトピックについてコードベースを調査し、
関連ファイル・関数・依存関係を構造化して報告する。
コードの変更は行わない。
```

**使い方：** 大きな機能を実装する前に、既存コードの調査を任せる。
メインのコンテキストを消費せずに情報収集できます。

### ② テストライター — テスト作成専門

```markdown
---
description: "テストコードの作成と実行"
allowed-tools:
  - read
  - write
  - bash
---

# テストライター
指定されたファイル・関数に対するテストを作成する。
- テストフレームワーク: Vitest
- 正常系・異常系・境界値の3パターン以上
- テスト実行して結果を報告する
```

### ③ セキュリティレビュワー — セキュリティ監査専門

```markdown
---
description: "セキュリティ上の問題点を検出する"
allowed-tools:
  - read
  - grep
  - glob
---

# セキュリティレビュワー
コードのセキュリティリスクを分析する。
チェック項目：
- SQLインジェクションの可能性
- XSS脆弱性
- 認証・認可の不備
- 機密情報のハードコーディング
- 依存パッケージの既知の脆弱性
重要度（CRITICAL/HIGH/MEDIUM/LOW）を付けて報告する。
```

#### ✅ 演習1：サブエージェントを1つ作る

1. `.claude/agents/` にリサーチャーの定義ファイルを作成
2. Claude Code で「このプロジェクトの認証周りのコードを調査して」と依頼
3. サブエージェントが起動してメインとは別のコンテキストで動くことを確認

## 4. Agent Teams — 並列処理

**注意：** この機能はまだ実験的で、環境変数で有効化が必要です。

```bash
export CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1
```

### Agent Teams とは

複数のサブエージェントを同時に起動し、チームとしてタスクに取り組む仕組みです。

### 協力パターン

```
「フロントエンドとバックエンドを並行して開発して」

→ Agent A: フロントエンド担当（React コンポーネント）
→ Agent B: バックエンド担当（API エンドポイント）
→ リードエージェント: 両者の整合性を調整
```

### 競争パターン

```
「この問題の解決策を2つのアプローチで実装して、比較して」

→ Agent A: アプローチ1で実装
→ Agent B: アプローチ2で実装
→ リードエージェント: 両者を比較して最適な方を選択
```

### 使いどころ

- フロントエンド + バックエンドの並行開発
- 複数ファイルの同時リファクタリング
- 異なるアプローチの比較検討
- 大規模な調査タスクの分割

## 5. Skills — 再利用可能な専門知識

Skills は、特定のタスクに対する「レシピ」です。
サブエージェントが「人」なら、Skills は「手順書」に近い概念です。

### Skills と他の仕組みの違い

| 仕組み | いつ読まれるか | 用途 |
|--------|-------------|------|
| CLAUDE.md | 毎セッション自動 | プロジェクトの基本情報 |
| rules/ | 毎セッション自動 | コーディング規約 |
| **skills/** | 必要なときだけ | 特定タスクの手順書 |
| agents/ | 呼び出されたとき | 専門エージェントの定義 |

### Skills の作り方

```
.claude/
└── skills/
    └── deploy/
        └── SKILL.md
```

```markdown
---
description: "本番環境へのデプロイ手順"
---

# デプロイ手順

## 事前チェック
1. 全テストが通ることを確認
2. lint エラーがないことを確認
3. main ブランチとの差分がないことを確認

## デプロイ手順
1. `npm run build` でビルド
2. `npm run deploy:staging` でステージングにデプロイ
3. ステージングで動作確認
4. `npm run deploy:production` で本番にデプロイ

## ロールバック
- 問題があれば `npm run rollback` で前のバージョンに戻す
```

**CLAUDE.md との違い：** Skills は「デプロイ時だけ」「レビュー時だけ」のように、
必要なときだけ読み込まれます。毎回のコンテキストを圧迫しません。

#### ✅ 演習2：Skill を1つ作る

1. 自分がよく行う定型作業を1つ選ぶ（デプロイ、レビュー、データ移行など）
2. `.claude/skills/<名前>/SKILL.md` にその手順を書く
3. 関連するタスクを依頼して、Skill が自動的に参照されるか確認

## 6. 全体の設計思想 — 最終整理

Phase 1〜6 で学んだ仕組みの全体像をまとめます。

### 「インデックスは軽く、中身は必要なときに読み込む」

これが全体を貫く設計原則です。

```
CLAUDE.md（30〜50行）
├── プロジェクトの概要（WHAT）
├── コマンド一覧（HOW）
├── 最重要ルール（TRAPS）
└── Skills への分岐表
    「デプロイ時は deploy スキルを使え」
    「レビュー時は review スキルを使え」

.claude/
├── rules/           ← 常に読み込まれるルール（規約）
├── skills/          ← 必要なときだけ読まれる手順書
├── agents/          ← 必要なときだけ起動する専門家
└── settings.json    ← 権限・フック・MCP 設定
```

### 判断フロー：どこに書くか

```
毎回必要な基本情報？ → CLAUDE.md
毎回必要なルール？   → .claude/rules/
特定タスクの手順？   → .claude/skills/
専門家として呼び出す？ → .claude/agents/
確実に実行させたい？  → hooks（settings.json）
外部ツール連携？     → MCP（settings.json）
```

## カリキュラム全体の振り返り

```
Phase 1: 基本操作         → ツールの「操縦方法」
Phase 2: CLAUDE.md        → Claudeに「プロジェクトを理解」させる
Phase 3: 指示の出し方     → 最高の出力を「引き出す」技術
Phase 4: Git連携          → 日常の開発フローに「組み込む」
Phase 5: MCP              → 外部ツールと「つなげる」
Phase 6: サブエージェント  → 複雑なタスクを「分業」させる
```

ここまで来たら、Claude Code を実務で使いこなす土台は完成です。
あとは日々の作業の中で経験を積み、自分なりのワークフローを確立してください。

公式のベストプラクティスにもこう書かれています：

> 何が機能するかに注意を払いましょう。
> Claude が素晴らしい出力を生成するとき、あなたが何をしたかに注目してください。
> 時間をかけて、ガイドが捉えることができない直感を開発しましょう。
