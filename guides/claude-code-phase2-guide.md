## 1. なぜ CLAUDE.md がこんなに重要なのか

### 技術的な理由を理解する

Claude Code は起動するたびに「記憶がゼロ」の状態から始まります。
前回のセッションで「TypeScriptで書いて」と伝えても、新しいセッションでは忘れています。

CLAUDE.md は、この問題を解決するファイルです。
セッション開始時に自動的に読み込まれ、「このプロジェクトはこういうものだ」という前提知識をClaudeに与えます。

### コンテキストウィンドウという制約

Claude には「コンテキストウィンドウ」という作業メモリがあります。
ここに入りきる情報量には上限があり、CLAUDE.md の内容もこの中に含まれます。

```
コンテキストウィンドウの中身：
┌──────────────────────────────┐
│ システムプロンプト            │ ← Claude自身の基本設定
│ CLAUDE.md の内容             │ ← 毎回読み込まれる
│ ツール定義                   │ ← ファイル編集等の機能
│ 会話履歴                     │ ← やりとりの蓄積
│ 読み込んだファイルの内容      │ ← @で指定したファイルなど
│ ─────────────────────────── │
│ ★ 残りが「考える余地」       │ ← ここが狭いと品質が落ちる
└──────────────────────────────┘
```

**つまり：CLAUDE.md が長すぎると、Claudeが考えるための余地が減り、指示を無視し始めます。**

## 2. 良い CLAUDE.md vs 悪い CLAUDE.md

### ❌ 悪い例：詰め込みすぎ

```markdown
# プロジェクト概要
これは○○のためのWebアプリケーションです。
ユーザーは△△を行うことができ、管理者は□□の機能を...
（中略、200行の詳細な仕様説明）

## コーディング規約
- インデントは2スペース
- セミコロンを使用する
- シングルクォートを使う
- 行末にスペースを入れない
- 最大行長は100文字
- import文は以下の順序で並べる：
  1. Node.js組み込みモジュール
  2. 外部パッケージ
  ...
（中略、50行のスタイルルール）

## API仕様
### GET /api/users
- パラメータ: page, limit, sort
- レスポンス: { users: [...], total: number }
（中略、全エンドポイントの詳細仕様100行）

## データベース設計
（テーブル定義50行）
```

**問題点：**
- 500行超で、Claudeがコンテキストを処理しきれない
- リンターで強制できるスタイルルールを書いている（ESLint/Prettierに任せるべき）
- API仕様やDB設計は、必要なときだけ参照すればいい情報

### ✅ 良い例：簡潔で実用的

```markdown
# ECサイト管理画面

React 19 + TypeScript strict + Next.js 15 (App Router)
DB: PostgreSQL (Prisma ORM)
スタイル: Tailwind CSS v4

## コマンド
- dev: `npm run dev`
- test: `npm test`
- lint: `npm run lint`
- build: `npm run build`

## 重要なルール
- APIレスポンスは必ずZodでバリデーションする
- 新しいページを作ったら必ずE2Eテストも追加する
- paymentsディレクトリのファイルは変更前に必ず確認を求める

## 注意点（罠）
- Prismaのマイグレーションは `npx prisma migrate dev` で実行（resetは絶対使わない）
- 環境変数は `.env.local` に追加したら `env.mjs` のスキーマも更新する
- next/imageのloaderは独自実装（S3直接参照）なので、デフォルトに戻さない
```

**良い点：**
- 30行程度で簡潔
- Claudeがコードから推測できない「罠」を明示
- リンターに任せられることは書いていない
- 本当に大事な禁止事項だけを書いている

## 3. 最初に書くべき3つの情報

CLAUDE.md にはまず以下の3つだけ書きましょう。それ以外は使いながら追加します。

### ① WHAT — このプロジェクトは何か

```markdown
# プロジェクト名

一行で目的を書く。
技術スタック: 言語、フレームワーク、DB、主要ライブラリをバージョン付きで列挙
```

「React プロジェクト」ではなく「React 19, Next.js 15.2」のようにバージョンまで書きます。
バージョンによってAPIが違うので、Claudeの出力精度に直結します。

### ② HOW — どうやってビルド・テスト・実行するか

```markdown
## コマンド
- 開発サーバー: `npm run dev`
- テスト: `npm test -- --watch`
- 型チェック: `npx tsc --noEmit`
- リント: `npm run lint`
```

Claudeはコードを書いた後、テストやリントで検証します。
正しいコマンドを知っていれば、自分で検証して品質を上げてくれます。

### ③ TRAPS — 罠と禁止事項

```markdown
## 注意点
- ○○ディレクトリのファイルは変更前に必ず確認を求める
- △△のライブラリはv3のAPIを使う（v2の書き方はNG）
- □□の処理は非同期にしない（理由：～）
```

**ここが一番大事です。** Claudeがコードから読み取れない「暗黙知」を書きます。
「これを書かなかったらClaudeが間違える」ものだけを厳選してください。

#### ✅ 演習1：3要素だけの CLAUDE.md を書く

1. 自分のプロジェクトで `claude` を起動
2. もし `/init` で生成済みなら内容を確認、なければ `/init` を実行
3. 生成された内容を見て、上の3カテゴリ（WHAT / HOW / TRAPS）に整理し直す
4. 不要な情報を削って、30行以内を目指す

## 4. メモリの階層構造 — どこに何を置くか

CLAUDE.md は実は複数の場所に置けます。
場所によって「誰に」「いつ」適用されるかが変わります。

### 配置場所と用途

```
~/.claude/CLAUDE.md             ← グローバル（全プロジェクト共通）
プロジェクト/CLAUDE.md          ← プロジェクト全体（チームで共有）
プロジェクト/CLAUDE.local.md    ← プロジェクト個人用（.gitignoreする）
プロジェクト/src/CLAUDE.md      ← サブディレクトリ固有
```

### 使い分けの考え方

| 配置場所 | 書く内容 | 例 |
|---------|---------|-----|
| `~/.claude/CLAUDE.md` | どのプロジェクトでも共通のルール | 「日本語で回答して」「計画を先に見せて」 |
| `プロジェクト/CLAUDE.md` | プロジェクト固有の情報 | 技術スタック、ビルドコマンド、コード規約 |
| `CLAUDE.local.md` | 個人の好みで、Gitに入れたくないもの | 「コミットメッセージは日本語で」 |
| `src/auth/CLAUDE.md` | 特定ディレクトリの注意点 | 「認証周りの変更は慎重に」 |

### まず作るべきは2つ

**Step 1：グローバル CLAUDE.md を作る**

```bash
# ファイルを作成
mkdir -p ~/.claude
nano ~/.claude/CLAUDE.md   # または好みのエディタ
```

```markdown
- 日本語で回答する
- 作業を始める前に計画を提示し、承認を得てから実装する
- コミットメッセージは feat: / fix: / chore: のプレフィックス付きで書く
- エラーハンドリングを省略しない
```

**Step 2：プロジェクト CLAUDE.md を整える**（演習1で作ったもの）

#### ✅ 演習2：グローバル CLAUDE.md を作る

1. `~/.claude/CLAUDE.md` を作成する
2. 上の例を参考に、全プロジェクト共通のルールを3〜5行で書く
3. Claude Code を起動して何かタスクを実行し、ルールが反映されるか確認

## 5. 日々の改善サイクル — こうやって育てる

CLAUDE.md は一度書いて終わりではなく、使いながら育てるものです。

### 育て方のフロー

```
Claudeが間違った動きをする
    ↓
「なぜ間違えたか」を考える
    ↓
コードから推測できない情報だった？
    ├── YES → CLAUDE.md に1行追加
    └── NO  → プロンプトの指示が曖昧だった（CLAUDE.mdの問題ではない）
```

### 追加するタイミングの判断基準

**CLAUDE.md に追加する（= 毎回必要な知識）：**
- 「テストコマンドが `npm test` じゃなくて `npm run test:unit` だった」
- 「このプロジェクトではdate-fnsではなくdayjsを使う」
- 「APIのレスポンスはsnake_caseで来るがフロントではcamelCaseに変換する」

**追加しない（= その場限りの指示）：**
- 「この関数のバグを直して」→ プロンプトで伝えればいい
- 「ここをリファクタリングして」→ 一時的な作業

### `/memory` コマンドで確認・編集

```
/memory
```

現在読み込まれているメモリファイルの一覧を確認し、その場で編集もできます。

### 自動メモリ機能

Claude Code は作業中に学んだことを自動でメモリに保存することがあります。
「このプロジェクトでは pnpm を使う」のようなパターンを自分で学習して記録します。

保存先は `~/.claude/projects/<プロジェクト>/memory/` で、
MEMORY.md がインデックスとして機能し、詳細は個別ファイルに保存されます。

#### ✅ 演習3：改善サイクルを回す

1. Claude Code で何かタスクを実行する
2. 出力結果で「惜しい、ここが違う」と思った点を見つける
3. それが「毎回言い直す必要がある」ことなら CLAUDE.md に1行追加
4. 別のタスクを実行して、追加したルールが反映されるか確認

## 6. 長くなったときの分割戦略

CLAUDE.md が肥大化したら（目安：200行超）、情報を外に切り出します。

### 分割先1：`.claude/rules/` — 常に読み込まれるルール

CLAUDE.md と同じ優先度で、セッション開始時に自動で読み込まれます。
トピックごとにファイルを分けて整理できます。

```
.claude/
├── CLAUDE.md              ← 概要・コマンド・最重要ルール（短く保つ）
└── rules/
    ├── code-style.md      ← コードスタイルの規約
    ├── testing.md         ← テストの書き方ルール
    └── security.md        ← セキュリティ要件
```

### 分割先2：`.claude/skills/` — 必要なときだけ読み込まれる

Skills は「必要なときだけ」読み込まれるのがポイントです。
毎回使わない手順書やワークフローはこちらに置きます。

```
.claude/
└── skills/
    └── deploy/
        └── SKILL.md       ← デプロイ手順（デプロイ時だけ読まれる）
```

### 分割先3：`@` で外部ファイルを参照

CLAUDE.md の中で `@path/to/file` と書くと、そのファイルの内容を取り込めます。
README や設計ドキュメントへのポインタとして使えます。

```markdown
# プロジェクト概要
詳細は @README.md を参照
利用可能なnpmコマンドは @package.json を参照
```

### 判断フロー：どこに書くべきか

```
この情報は「毎回のセッション」で必要？
├── YES → 50行以内に収まる？
│   ├── YES → CLAUDE.md に書く
│   └── NO  → .claude/rules/ に分割
└── NO → 特定のタスクでだけ必要？
    ├── YES → .claude/skills/ に置く
    └── NO  → docs/ に置いて、必要なとき @ で参照
```

#### ✅ 演習4：分割を試す（CLAUDE.md が長くなった人向け）

1. 現在の CLAUDE.md を見て、200行を超えていないか確認
2. 超えていたら、「毎回必要」と「たまに必要」に分類
3. 「毎回必要」は `.claude/rules/` に分割
4. 「たまに必要」は `.claude/skills/` に移動
5. CLAUDE.md 本体を30〜50行に縮小

## 7. よくある落とし穴と対策

### 落とし穴1：リンターで強制できることを書いている

```markdown
# ❌ これは書かない
- インデントは2スペース
- セミコロンを使用する
- 未使用のimportは削除する
```

→ ESLint / Prettier の設定ファイルがあればClaudeは自動で検出します。
CLAUDE.md に書くのは **ツールで強制できないルール** だけです。

### 落とし穴2：曖昧な指示を書いている

```markdown
# ❌ 曖昧
- コードを適切にフォーマットする
- テストをちゃんと書く
- パフォーマンスを意識する

# ✅ 具体的
- 全てのAPI関数にはZodスキーマによる入力バリデーションを書く
- 新しいページを追加したら、対応するE2Eテストを tests/e2e/ に追加する
- リスト表示は100件を超えるならページネーションを実装する
```

### 落とし穴3：定期的に見直していない

CLAUDE.md はコードと同じで、プロジェクトの変化に合わせて更新が必要です。
月に1回は読み返して「まだ有効か」を確認しましょう。

削除の判断基準：**「この行を消してもClaudeは間違えないか？」**
→ 消しても大丈夫なら、消す。短いほど効果的です。
