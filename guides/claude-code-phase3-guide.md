# Phase 3：指示の出し方を磨く

> 目標：Claude Code から最高の出力を引き出すプロンプトの書き方を身につける
>
> 所要時間：1〜2週間（日々の作業で実践しながら）
>
> 前提：Phase 1（基本操作）と Phase 2（CLAUDE.md）が完了している

---

## このフェーズで身につけること

1. 「計画 → 実装」の2段階ワークフロー
2. タスクを分割して指示するテクニック
3. 拡張思考（think / ultrathink）の使い分け
4. 良い指示 vs 悪い指示の具体例
5. コンテキスト管理 — 会話を効率的に保つ方法
6. フィードバックと軌道修正のやり方

## 1. 最重要原則：「まず計画、それから実装」

Claude Code で最もよくある失敗は、「大きなタスクをいきなりやらせる」ことです。

### なぜ失敗するのか

Claude が個々の判断で正しい選択をする確率が仮に80%だとしても、
20箇所の判断がある機能では、すべて正しい確率は約1%にまで下がります。
計画を先に立てることで、曖昧な判断を「レビュー済みの方針」に置き換えられます。

### 基本ワークフロー

```
Step 1: Plan mode で計画を出させる（Shift+Tab ×2）
    ↓ 計画を読んで、問題があれば修正指示
Step 2: Normal mode に戻して実装させる（Shift+Tab）
    ↓ 変更を /diff で確認
Step 3: テストを実行して検証
```

### 実践例

```
❌ 悪い例（いきなり全部やらせる）：
「ユーザー認証機能を実装して」

✅ 良い例（計画から入る）：
「ユーザー認証機能を追加したい。
 まず実装計画を立てて。コードはまだ書かないで。
 以下を含めて：
 - 必要なファイルと変更箇所
 - 使用するライブラリ
 - セキュリティ上の考慮点」
```

計画が返ってきたら内容を確認して、問題がなければ「この計画で進めて」と指示します。
計画に問題があれば「Step 2のセッション管理はJWTではなくcookieベースにしたい」のように修正します。

#### ✅ 演習1：計画 → 実装のフローを体験する

1. `claude` を起動し、`Shift+Tab` ×2 で Plan mode に入る
2. 何かタスクを伝える（例：「TODOアプリにカテゴリ機能を追加したい」）
3. 計画を読んで、1つ以上の修正フィードバックを返す
4. 修正された計画に納得したら、`Shift+Tab` で Normal mode に戻す
5. 「この計画で進めて」と指示して実装させる

## 2. タスク分割 — 大きな仕事を小さく割る

### 1セッション1タスクの原則

Claude Code の会話が長くなるほど、コンテキストがノイズで溢れ、品質が落ちます。
目安として、コンテキスト使用率が60%を超えると指示の見落としが増え始めます。

**1つのセッションでは1つの明確なタスクだけをやる。
終わったら `/clear` して次のタスクに移る。**

### 分割の考え方

```
「ECサイトのカート機能を実装して」
    ↓ 分割すると
Session 1: 「カートのデータモデルとAPI設計を作って」
Session 2: 「カート追加・削除のAPIを実装して」
Session 3: 「カート画面のUIコンポーネントを作って」
Session 4: 「カート機能のテストを書いて」
```

各セッションの冒頭で、前のセッションで作ったものの概要を簡潔に伝えます。
CLAUDE.md にプロジェクトの全体像があれば、最小限の説明で済みます。

### 段階的に進める指示の出し方

1つのセッション内でも、ステップを分けて指示すると精度が上がります。

```
Step 1: 「まず @src/models/cart.ts を見て、現在のデータ構造を把握して」
Step 2: 「この構造に合うバリデーション関数を追加して」
Step 3: 「追加した関数のテストを書いて」
Step 4: 「テストを実行して、結果を教えて」
```

#### ✅ 演習2：タスク分割を実践する

1. 少し大きめの機能を思い浮かべる
2. それを3〜5のサブタスクに分割する
3. 最初のサブタスクだけを1セッションで実行
4. `/clear` して次のサブタスクに移る
5. 分割しなかった場合と比べて、出力の質を比較する

## 3. 拡張思考 — think / ultrathink の使い分け

Claude Code にはプロンプト内のキーワードで思考の深さを制御する仕組みがあります。

### 思考レベル一覧

| キーワード | 思考トークン量 | コスト | 使いどころ |
|-----------|-------------|------|-----------|
| (指定なし) | 通常 | 低 | 単純な質問、小さな修正 |
| `think` | 少量 | 中 | 軽い設計判断 |
| `think hard` / `think harder` | 中量 | 中〜高 | リファクタリング戦略、設計判断 |
| `ultrathink` | 最大（約31,999トークン） | 高 | アーキテクチャ設計、複雑なバグ調査 |

### いつどれを使うか

**ultrathink が効果的な場面：**
- 複数モジュールが絡む複雑なバグの原因調査
- アプリ全体のアーキテクチャ設計
- 大規模なリファクタリングの方針決定
- 正解が複数あり得る設計判断

**ultrathink を使うべきでない場面：**
- 「このボタンの色を変えて」のような単純な変更
- 定型的なCRUD実装
- typoの修正
- テストの追加

### 注意点

- ultrathink はトークンを大量に消費するので、本当に必要な場面だけに使う
- Plan mode + ultrathink が最強の組み合わせ（計画段階で深く考えさせる）
- 拡張思考モードでは、細かすぎるステップ指示は逆効果。
  「ステップ1で○○して、ステップ2で△△して」より
  「徹底的に考えて、複数のアプローチを検討して」の方が良い結果が出る

### 実践例

```
# 軽い作業 — think 不要
「この関数の戻り値の型を修正して」

# 中程度の設計判断 — think hard
「このAPIの認証方式を think hard で検討して。
 JWT vs セッションベース、それぞれのメリデメを比較して」

# 重大な判断 — ultrathink
「データベースのマイグレーション戦略について ultrathink で考えて。
 現在のスキーマから新しいスキーマへの移行パスを検討して。
 ダウンタイムなしで移行する方法を提案して」
```

#### ✅ 演習3：思考レベルの違いを体感する

1. 同じ設計系の質問を、指定なし → think hard → ultrathink で試す
2. 回答の深さ・網羅性がどう変わるか比較する
3. 「この質問にはどのレベルが最適だったか」を振り返る

## 4. 良い指示 vs 悪い指示 — 具体例集

### パターン1：曖昧 vs 具体的

```
❌ 「このコードを改善して」

✅ 「この関数のパフォーマンスを改善して。
   具体的には、配列の処理がO(n²)になっている部分を
   O(n)にしたい。HashMapの使用を検討して」
```

### パターン2：ゴールが不明 vs ゴールが明確

```
❌ 「APIを作って」

✅ 「ユーザー一覧を取得するGET /api/usersエンドポイントを作って。
   要件：
   - ページネーション対応（page, limit パラメータ）
   - レスポンスは { users: [...], total: number } の形式
   - Zodでリクエストパラメータをバリデーション
   - エラーハンドリングを含める」
```

### パターン3：制約なし vs 制約あり

```
❌ 「テストを書いて」

✅ 「@src/services/auth.ts のテストを書いて。
   - テストフレームワーク: Vitest
   - モックは vi.mock で作る
   - 正常系・異常系・境界値の3パターン以上
   - テストファイルは tests/services/auth.test.ts に配置」
```

### パターン4：用途を伝えない vs 用途を伝える

```
❌ 「このデータを表にまとめて」

✅ 「このデータを表にまとめて。
   これは来週の経営会議で使う資料なので、
   重要な数値とトレンドが一目で分かるようにして」
```

用途を伝えるだけで、Claude は出力のスタイルと深さを適切に調整します。

### パターン5：否定形 vs 肯定形

```
❌ 「console.logを使わないで」

✅ 「ログ出力にはプロジェクトのloggerモジュール（@src/lib/logger.ts）を使って」
```

「やるな」より「代わりにこうしろ」の方が、Claudeは正確に従います。

#### ✅ 演習4：自分の指示を改善する

1. 最近 Claude Code に出した指示を思い出す
2. 上のパターンに当てはめて「改善版」を書く
3. 改善した指示で再度実行して、出力の違いを比較する

## 5. コンテキスト管理 — 会話を効率的に保つ

### `@` でファイルを直接指定する

Claude にファイルを探させるのはトークンの無駄です。
直接指定すれば、探索なしで的確な回答が得られます。

```
# ❌ ファイルを探させる
「ヘッダーコンポーネントを修正して」

# ✅ 直接指定する
「@src/components/Header.tsx のナビゲーション部分を
 レスポンシブ対応に修正して」
```

### `!` でシェルコマンドの結果を渡す

テスト結果やエラーログをClaudeに見せたいとき、
`!` で実行すれば結果が自動的にコンテキストに入ります。

```
! npm test                    # テスト結果を見せてから
「失敗しているテストを修正して」

! git diff                    # 変更差分を見せてから
「この変更のコードレビューをして」
```

### `/compact` と `/clear` の使い分け（復習）

```
同じタスクの続き → /compact （要約して圧縮、文脈を保持）
別のタスクに移る → /clear  （完全リセット）
```

`/compact` には引数を渡して、残す情報を指定できます。

```
/compact APIの設計方針とエラーパターンだけ残して
```

### スクリーンショットを活用する

UIのバグ修正やデザイン実装では、スクリーンショットを直接渡すのが最速です。

```
Ctrl+V（Mac）/ Alt+V（Windows/Linux）で画像をペースト

「この画面の赤枠部分のレイアウトが崩れている。修正して」
```

エラー画面、デザインモックアップ、手書きの設計図まで、
画像をそのまま渡して指示できます。

#### ✅ 演習5：コンテキスト管理を実践する

1. `@` でファイルを指定して質問する
2. `! npm test` の結果を使って修正を依頼する
3. 会話が長くなったら `/compact` で圧縮して、品質が保たれるか確認

## 6. フィードバックと軌道修正

Claudeの出力が期待と違ったとき、どう立て直すかも重要なスキルです。

### 出力が方向違いだったとき

```
# 途中で止めて方向修正
Ctrl+C で出力を中断

「方向性が違う。○○ではなく△△のアプローチで進めて。
 理由は□□だから」
```

待つ必要はありません。違うと思った瞬間に `Ctrl+C` で止めましょう。

### 出力の一部だけ修正したいとき

```
「いま作ったコードの大部分はOK。ただし以下を修正して：
 1. エラーハンドリングが try-catch で握りつぶしている → 適切にthrowする
 2. ログメッセージが英語 → 日本語に変更」
```

何が良くて何が悪いかを具体的に伝えることで、
必要な部分だけを修正し、良い部分はそのまま残してくれます。

### 同じ間違いを繰り返すとき

Claudeが同じ間違いを繰り返す場合、2つの対処法があります。

**対処法1：プロンプトを見直す**
指示が曖昧だった可能性があります。より具体的に書き直します。

**対処法2：CLAUDE.md に追記する**
毎回言い直す必要があるなら、それはCLAUDE.md に書くべきルールです。
（Phase 2 で学んだ改善サイクル）

### Claudeが詰まったとき

ループして同じ修正を繰り返している場合は、思考レベルを上げます。

```
「別のアプローチで考え直して。ultrathink で。
 現在のアプローチがうまくいかない理由も分析して」
```

## 7. 実践的なワークフロー例

### 新機能を追加する場合

```
1. Plan mode + think hard で計画立案
   → 計画をレビュー、修正フィードバック

2. /clear して Normal mode で実装開始
   → 計画の Step 1 から順に指示

3. 各ステップ完了後に ! npm test で検証
   → 失敗したら @失敗したテストファイル を指定して修正依頼

4. 全体完了後に /diff で変更確認
   → 問題なければ /commit でコミット
```

### バグを修正する場合

```
1. エラーログやスクリーンショットを貼り付けて状況を伝える
   → 「このエラーの原因を調査して。think hard で。」

2. 原因の仮説が返ってきたら内容を確認
   → 「この仮説で合ってる。修正して」or「別の可能性も検討して」

3. 修正後にテスト実行
   → ! npm test の結果を確認
```

### コードレビューを依頼する場合

```
「@src/services/payment.ts をレビューして。
 特に以下の観点で：
 - セキュリティ上の問題がないか
 - エラーハンドリングが適切か
 - テストが不足している箇所はどこか」
```
