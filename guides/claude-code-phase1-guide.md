## 1. 起動と終了 — まずここから

### 起動

```bash
# プロジェクトのフォルダに移動してから起動する
cd ~/my-project
claude
```

**重要ポイント：** Claude Codeは「どのフォルダで起動したか」を見て、そのプロジェクト全体を自動的にスキャンします。必ず作業したいプロジェクトのルートディレクトリで起動してください。

### 終了

```
# セッション内で入力
/exit
```

または `Ctrl+D` でも終了できます。

### セッション再開（超重要！）

Claude Codeは、過去のセッション（会話）を記憶しています。
前回の続きをやりたいときに毎回ゼロから説明し直す必要はありません。

```bash
# 直前のセッションを再開
claude --continue
# または短縮形
claude -c

# 過去のセッション一覧から選んで再開
claude --resume
# または短縮形
claude -r
```

#### ✅ 演習1：起動 → 終了 → 再開を体験する

1. 適当なフォルダで `claude` を起動
2. 「こんにちは、テストです」と入力
3. `/exit` で終了
4. `claude -c` で再開 — 先ほどの会話が残っているか確認

## 2. 3つの操作モード — Shift+Tab で切り替え

Claude Codeには3つのモードがあり、`Shift+Tab` を押すたびに切り替わります。
画面下部にどのモードかが表示されます。

### モード一覧

| モード | 動作 | いつ使う？ |
|--------|------|-----------|
| **Normal**（デフォルト） | ファイル変更のたびに「OK？」と確認が入る | 初めての作業、慎重に進めたいとき |
| **Auto-accept** | ファイル編集を自動で実行（確認なし） | やり方が分かっている定型作業 |
| **Plan mode** | 計画だけ立てて、実際のファイル変更はしない | 複雑な作業を始める前の設計段階 |

### 切り替え方

```
Shift+Tab 1回目 → Auto-accept に切り替え
Shift+Tab 2回目 → Plan mode に切り替え
Shift+Tab 3回目 → Normal に戻る
```

### 各モードのポイント

**Normal モード（最初はこれでOK）**
- ファイルの作成・変更のたびに確認を求められる
- 「y」で承認、「n」で却下
- 何をされるかひとつひとつ確認できて安心

**Auto-accept モード（慣れてきたら）**
- ファイル編集が全自動になるので高速
- ただしコマンド実行（npm installなど）は引き続き確認あり
- 「型定義を全ファイルに追加して」のような一括作業向き
- 作業後は `/diff` で変更内容を必ず確認する癖をつける

**Plan mode（一番重要かもしれない）**
- Claude が「何をするか」の計画だけ出し、実行はしない
- 計画を見て「OK」と思ったら Normal に戻して実行させる
- 「ログイン機能を追加して」のような大きなタスクは必ず Plan mode から始める

#### ✅ 演習2：3つのモードを体感する

1. `claude` を起動
2. Normal モードのまま「READMEファイルを作成して」→ 確認が来ることを体感
3. `Shift+Tab` で Auto-accept に切り替え → 画面下部の表示を確認
4. `Shift+Tab` で Plan mode に切り替え → 「簡単なTODOアプリの構成を考えて」と入力
5. 計画だけ出力されてファイルが作られないことを確認
6. `Shift+Tab` で Normal に戻す

## 3. 最初に覚えるスラッシュコマンド7つ

セッション中に `/` を入力すると候補一覧が表示されます。
まずはこの7つだけ覚えれば日常作業に困りません。

### 3-1. `/help` — 困ったらまずこれ

```
/help
```
使えるコマンドの一覧が表示されます。何か分からなくなったらとりあえずこれ。

### 3-2. `/init` — プロジェクトの初期設定

```
/init
```
プロジェクトのコードをスキャンして、CLAUDE.md というファイルを自動生成します。
このファイルが「プロジェクトの取扱説明書」になり、以降のセッションでClaudeの理解度が大幅に上がります。

**生成される CLAUDE.md の中身（例）：**
- プロジェクトで使っている言語・フレームワーク
- ビルドコマンド、テスト実行方法
- ディレクトリ構成の説明

### 3-3. `/clear` — 会話をリセット

```
/clear
```
会話履歴を全消去して、まっさらな状態にします。
**別のタスクに切り替えるとき**に使います。

### 3-4. `/compact` — 会話を圧縮

```
/compact
```
会話が長くなると、Claudeの理解力が落ちたりトークン（利用量）を消費しすぎます。
`/compact` は会話を要約して短くしつつ、重要な情報は保持してくれます。

```
# 特定の情報を残したいとき
/compact APIの設計方針だけ残して
```

### `/compact` と `/clear` の使い分け

| 状況 | 使うコマンド |
|------|------------|
| 同じタスクを続けるが会話が長くなった | `/compact` |
| 別のタスクに切り替える | `/clear` |
| コンテキスト使用率が80%超 | まず `/compact`、改善しなければ `/clear` |

### 3-5. `/diff` — 変更内容を確認

```
/diff
```
Claudeが行ったファイル変更の差分をビジュアルに確認できます。
左右矢印キーでgit diffと個別ターンのdiffを切り替え、上下キーでファイルを移動できます。

**Auto-acceptモードで作業した後は必ず `/diff` で確認する**のが安全な運用です。

### 3-6. `/status` — 現在の状態確認

```
/status
```
アカウント状態、使用モデル、利用状況などを確認できます。
「あれ、今日もう使えない？」と思ったらこれで確認。

### 3-7. `/model` — モデル切り替え

```
/model
```
使用するAIモデルを切り替えます。

- **Opus** — 最も賢い。複雑な設計・実装向き
- **Sonnet** — 速くてコスパが良い。日常的な作業向き

簡単な質問やリファクタリングは Sonnet、複雑な機能追加は Opus、のように使い分けるとトークンを節約できます。

#### ✅ 演習3：スラッシュコマンドを一通り試す

1. プロジェクトフォルダで `claude` を起動
2. `/help` でコマンド一覧を確認
3. `/init` を実行 → CLAUDE.md が生成されるか確認
4. 何か簡単な指示を出す（例：「このプロジェクトの構成を説明して」）
5. `/compact` で会話を圧縮
6. `/status` で状態確認
7. `/clear` で会話をリセット

## 4. 覚えるべきキーボードショートカット5つ

### 最重要の5つ

| ショートカット | 動作 | 使いどころ |
|--------------|------|-----------|
| `Ctrl+C` | Claudeの応答を中断 | 方向性がおかしいと思ったら即止める |
| `Shift+Tab` | モード切り替え | Normal ↔ Auto-accept ↔ Plan |
| `Esc` × 2回 | 変更の巻き戻し（undo） | Claudeが変なことをしたとき |
| `@` | ファイルパス補完 | 特定ファイルをClaudeに見せたいとき |
| `!` | シェルコマンド直接実行 | `! git status` のように使う |

### `Ctrl+C` — 一番大事

Claudeの出力が長すぎる、方向性が違う、と思った瞬間に `Ctrl+C` で止められます。
セッションは終了しないので、止めた上で指示をやり直せます。
**待つ必要はありません。**違うと思ったらすぐ止める癖をつけてください。

### `@` — ファイルを直接指定

```
@src/components/Header.tsx このファイルのレスポンシブ対応を修正して
```

`@` を入力するとファイル名の補完候補が出ます。
Claudeにファイルを探させるより、直接指定した方がトークン節約にもなり、的確な回答が得られます。

### `!` — シェルコマンドをそのまま実行

```
! git status
! npm test
! ls -la src/
```

先頭に `!` をつけると、Claudeを介さずにコマンドを直接実行できます。
結果はClaudeの会話コンテキストに追加されるので「テスト結果を見て修正して」のような流れが作れます。
トークンを消費しないので、ちょっとした確認にはこちらを使いましょう。

#### ✅ 演習4：ショートカットを試す

1. `claude` を起動して何か長い指示を出す
2. 出力途中で `Ctrl+C` で止めてみる → セッションが続いていることを確認
3. `@` を入力してファイル補完を体験
4. `! pwd` でカレントディレクトリを確認
5. `! git status` でGitの状態を見る（Gitリポジトリの場合）

## 5. `/init` と CLAUDE.md — なぜ重要か

### CLAUDE.md とは

CLAUDE.md は、Claude Codeが起動時に必ず読み込む「プロジェクトの設定ファイル」です。
ここに書かれた内容に基づいて、Claudeは回答の精度を上げます。

### 置き場所

```
my-project/
├── CLAUDE.md          ← プロジェクト全体の設定（チームで共有）
├── src/
│   └── CLAUDE.md      ← サブディレクトリ固有の設定（任意）
└── ...
```

### `/init` で何が起きるか

1. プロジェクト内のファイルをスキャン
2. 使用言語、フレームワーク、ビルドコマンドなどを自動検出
3. CLAUDE.md のドラフトを生成
4. 「この内容でOK？」と確認される

### CLAUDE.md に書くとよい情報

```markdown
# プロジェクト概要
○○をするWebアプリケーション

## 技術スタック
- フロントエンド: React + TypeScript
- バックエンド: Node.js + Express
- DB: PostgreSQL

## コマンド
- ビルド: `npm run build`
- テスト: `npm test`
- 開発サーバー: `npm run dev`

## コーディング規約
- 変数名はキャメルケース
- コンポーネントファイルはPascalCase
- コミットメッセージは日本語OK
```

**Phase 2 でこのファイルを充実させていきますが、Phase 1 ではまず `/init` で生成するところまでやっておきましょう。**

#### ✅ 演習5：CLAUDE.md を生成する

1. 既存のプロジェクト（なければ簡単なプロジェクトを作成）のルートで `claude` を起動
2. `/init` を実行
3. 生成された CLAUDE.md の内容を読む
4. 実態と合っているか確認する（合っていなくても Phase 2 で修正するので今はOK）

## 6. 複数行入力 — 長い指示を出すコツ

1行だけでは伝えきれない指示を出したいとき、改行して複数行で入力できます。

### 改行方法

| 方法 | 対応環境 |
|------|---------|
| `\` を行末に置いて Enter | すべてのターミナル |
| `Shift+Enter` | iTerm2, WezTerm, Ghostty, Kitty |

※ `Shift+Enter` が使えない環境では、セッション内で `/terminal-setup` を実行するとバインディングがインストールされます。

### 複数行入力の例

```
ログインフォームを作成してください。\
以下の要件で：\
- メールアドレスとパスワードのフィールド\
- バリデーション付き\
- エラーメッセージの表示\
- React + TypeScript で実装
```

## 7. 拡張思考 — 「もっと深く考えて」とお願いする

複雑な問題でClaudeにしっかり考えてほしいとき、プロンプトに特定のキーワードを入れると「拡張思考モード」が発動します。

| キーワード | 思考の深さ | 使いどころ |
|-----------|-----------|-----------|
| `think` | 軽い思考 | ちょっと考えてほしいとき |
| `think hard` | 中程度 | 設計判断が必要なとき |
| `ultrathink` | 最大（トークン消費大） | アーキテクチャの大きな判断 |

### 使い方の例

```
このアプリのデータベース設計を考えて。ultrathinkで。
```

```
このエラーの原因を分析して。think hardで。
```

**注意：** ultrathink はトークンを大量に消費するので、本当に重要な判断のときだけ使いましょう。日常的な作業には不要です。
