## 1. 自然言語で Git を操作する

Claude Code は Git コマンドをそのまま自然言語で指示できます。
MCP 設定なしでも、ローカルの Git 操作はすべて対応しています。

### 基本操作

```
「新しいブランチ feature/user-auth を作って」
→ git checkout -b feature/user-auth を実行

「今の変更状況を教えて」
→ git status を実行して結果を説明

「全ての変更をステージングして」
→ git add . を実行
```

### `!` で直接実行との使い分け

```
# Claude に解釈させたいとき（状況説明付き）
「今の変更状況を教えて」

# 単にコマンドを実行したいとき（トークン節約）
! git status
! git log --oneline -5
```

単純な確認は `!` で直接実行し、判断が必要な操作は自然言語で指示するのが効率的です。

## 2. コミットメッセージの自動生成

```
「この変更をコミットして」
```

これだけで、Claude が変更差分を分析し、Conventional Commits 形式のメッセージを自動生成してくれます。

```
例：
feat: ユーザー認証エンドポイントを追加
fix: ログインフォームのバリデーションエラーを修正
chore: 依存パッケージを更新
```

### コミットメッセージのルールを CLAUDE.md に書いておく

```markdown
## Git 規約
- コミットメッセージは feat: / fix: / chore: / docs: のプレフィックス付き
- 本文は日本語で書く
- 1コミット1機能の原則を守る
```

これを CLAUDE.md に書いておけば、毎回同じスタイルで生成されます。

### `/diff` で変更を確認してからコミット

```
/diff              ← 変更内容を目視確認
「この変更をコミットして」  ← 問題なければコミット
```

#### ✅ 演習1：コミットの自動化を体験する

1. ファイルを何か変更する
2. `/diff` で変更を確認
3. 「この変更をコミットして」と指示
4. 生成されたコミットメッセージの内容を確認

## 3. ブランチ作成 → 実装 → PR の一気通貫フロー

Claude Code を使った開発の典型的なフローです。

```
Step 1: 「feature/add-search ブランチを作って」
Step 2: Plan mode で計画立案
Step 3: Normal mode で実装
Step 4: 「テストを実行して」→ ! npm test
Step 5: 「この変更をコミットして」
Step 6: 「プッシュして PR を作って」
```

Step 6 では GitHub CLI（`gh` コマンド）がインストールされていれば、
PR の作成まで自動で行われます。PR の説明文も変更内容に基づいて自動生成されます。

### GitHub CLI がない場合

```
「この変更をプッシュして」
→ git push origin feature/add-search

# PR は GitHub の Web UI から手動で作成
```

#### ✅ 演習2：ブランチ → コミット → プッシュを一連で行う

1. 新しいブランチを作らせる
2. 簡単な変更を実装させる
3. コミットさせる
4. プッシュさせる（GitHub リポジトリがある場合）

## 4. `/review` でコードレビューを依頼する

```
/review
```

PR の URL を指定するか、現在のブランチの変更をレビューしてもらえます。

### レビュー観点を指定する

```
「@src/services/payment.ts をレビューして。
 以下の観点で：
 - セキュリティ上の問題
 - エラーハンドリングの抜け漏れ
 - テストが不足している箇所」
```

### 他人が書いたコードのレビュー

```
「PR #42 の変更をレビューして。
 特にデータベースクエリのパフォーマンスに注目して」
```

### レビュー結果の活用

レビューで指摘された問題をそのまま修正させられます。

```
「いまのレビューで指摘された問題のうち、
 CRITICAL と HIGH のものを修正して」
```

#### ✅ 演習3：コードレビューを試す

1. 何か変更を加えた後、`/review` を実行
2. 指摘された内容を読む
3. 「この指摘を修正して」と依頼して修正させる

---

## 5. コンフリクト解消

マージ時にコンフリクトが発生しても、Claude に解決を任せられます。

```
「main ブランチをマージして、コンフリクトがあれば解決して」
```

Claude は両方の変更内容を読み取り、適切な解決策を提案・実行します。
ただし、ビジネスロジックの判断が必要な場合は自分で確認しましょう。

## 6. GitHub Actions 連携（概要）

より高度な連携として、PR 作成時に自動で Claude にレビューさせる仕組みがあります。

### 仕組み

1. PR が作成される → GitHub Actions がトリガー
2. Claude Code Action が差分を分析
3. バグの可能性、セキュリティの懸念、スタイル違反を自動コメント
4. 人間のレビュアーは Claude の指摘を参考にしつつ、設計判断に集中

### セットアップ

```bash
# Claude Code のセッション内で実行
/install-github-app
```

このコマンドで必要な GitHub App とシークレットの設定が行われます。

**個人開発では必須ではないので、チーム開発に移行するタイミングで導入を検討してください。**
